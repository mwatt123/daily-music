# Add a Preference-Signal-driven re-curation script, despite issue #1's "no curation tooling" scope note

The original 434-album curation (commit `4ec3327`) picked albums per artist using an arbitrary cap-of-3, with no signal for which album was actually preferred — several favorites were dropped entirely, and for artists with multiple surviving albums, the wrong one often won. We're introducing a reusable, in-repo script that parses an Apple Music `Library.xml` export into per-track play counts, aggregated into an average-per-track Preference Signal per album, driving a new cap-of-1 (Definitive Album) selection rule, with a small hand-maintained override list for cases where taste diverges from listening frequency.

This is a deliberate exception to issue #1's "Admin/curation tooling ... out of scope" note — that exclusion was about end-user-facing curation UI, not a developer-only re-sync script. The script does mechanical extraction only, matching `(Artist, Album)` exactly with no fuzzy merging of near-duplicate titles (e.g. remasters); classification (studio album vs. single/EP/live/compilation/reissue, and reissue/remaster dedup) stays an LLM judgment call, since Apple Music's own classification metadata was already found unreliable during the original curation pass. The raw `Library.xml` export is never committed — it's personal listening data, regenerated locally each time the script runs.

**Consequences**: pipeline output is diffed against the current list rather than applied wholesale; high-confidence changes (clear Preference Signal winner, unambiguous classification) apply directly, ambiguous ones (near-tied signal, unclear classification) are surfaced for manual review instead of being applied blindly.
